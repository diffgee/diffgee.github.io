Adding the GroupRadio feature to YouTube Music

Building user delight through product iteration

Context

Listening to music in social settings is an extremely popular and common behavior. Many of the solutions offered by music streaming services for this context leave users with either impersonal experiences, or require a great deal of user effort. In this project, I design a satisfying and frictionless way for groups to enjoy music together.

Outcome

The GroupRadio feature designed for YouTube Music allows groups of all sizes to effortlessly build a radio station which keeps the whole group happy.

Research

Goals

I wanted to understand how and why users might listen to music in a social setting. In better understanding users behavior and desires, I aimed to continue the design process with data-informed decision making.

Objectives

  • Determine how people currently use music streaming products
  • Learn what goals the users are aiming to accomplish
  • Identify what barriers users face in their journey

Methods

  • Secondary research
  • Competitive analysis
  • User Journey
  • User Persona

Secondary Research and Competitive Analysis

One of the key findings of this research is that people frequently engage in listening to music in a social setting.

I honed in on this finding, and began researching how current products address this common user behavior.

During my product comparison research I created a document identifying the products and their key features.

Using these findings I identified one product in particular, Spotify Blend, as being highly relevant to this new potential feature to YouTube Music.

Briefly, Spotify Blend is a feature which allows two users to share a playlist. This playlist is created by an algorithm which looks at each users listening habits and attempts to ‚Äúblend‚Äù their interests together into a single music source.

I wanted to learn more about the user experience of this feature.

  • Did people enjoy it? Why?
  • Is there something that detracts from their experience when using Blend?
  • Are there requests for expanded function?

Understanding the User

In order to further define the feature, I wanted to gain deeper insight into the needs of a user. I relied on three main deliverables to achieve this:

  • 4W‚Äôs and a POV statement (who, what, where, why)
  • User persona
  • User story scenario

Research Conclusion

Now that I had a deeper understanding of the user, I was able to define a specific problem users face.

People enjoy listening to music in groups, but struggle to easily find common ground.

Ideate

Finding a Solution

In order to identify features which would delight users, I created a series of How Might We statements, and possible solutions to bring those features to life.

I then created a user scenario. I wanted to further define the feature through a user centered view. I wanted to communicate the user problem, user need, and solution provided by the feature in a succinct manner.

Culling from the generated ideas, a minimum viable product (MVP) feature set was determined.

Product Features

  • A sharable radio station
  • Streamlined start, share, and join
  • Leverage metadata and AI to handle details of music curation, allowing users to focus on their social setting

Design

In order to communicate how the feature would be used, I created a user flow. This started as a flowchart, and in a later iteration product screens were incorporated to better display the process.

I named the feature GroupRadio to communicate the social nature of the feature, and radio-like setup (ie. low effort , auto-curated music start).

User Interface

My main goals while designing the screens were:

  • Disruption to the existing product should be minimal. The feature addition should not be a lengthy overhaul of a product people are already familiar with.

  • The features behavior and look needs to fit existing design system. Users of the existing product should feel at home with the behavior of interactions and symbols.

  • The steps for use need to be minimal. A key finding during the research phase was the prominence of ‚Äúlow effort‚Äù music listeners. The perfect radio would have one button while providing a delightful user experience.


Where possible, I made use of existing resources and screens. When new assets and interactions were designed, I conformed them to mesh with the existing product.

Validate

User Testing

In order to user test and validate the design, I recruited three individuals. Two were current users of YouTube Music, while one was unfamiliar with it. Individual testing sessions were conducted in person. I described the feature, a brief scenario, and asked the user to start a new GroupRadio. Upon completion of that task, I asked them to go through the process of adding another user.

Results

The results of this testing session were mixed. Most users struggled to find the way to start the GroupRadio. However, the second task of adding a user was completed by all users without further difficulty. Additionally, even upon successful task completion users displayed confusion on the details of how the feature functioned, and how it differed from existing features such as a shared playlist.

Iterate

The confusion regarding the nature of the feature is understandable. The differences between features like auto-generated playlists, streaming radio, and shared playlists are subtle but meaningful. Often, user interfaces might make use of tooltips to inform users of functionality. It hasn‚Äôt been a traditional approach of the YouTube Music app, nor did I feel it would give sufficient context to the user on fully understanding how this feature is distinct from others.

Youtube Music often highlights new features with emailed newsletters and landing pages. I decided to design a product feature landing page which could be used in this manner. This approach allowed me to succinctly display the product feature, it‚Äôs likely use cases, and how to use it- all in a way that was on-brand and undisruptive of the existing app experience.

Designer Insights

In this case study I designed a feature addition, GroupRadio, for an existing product using the Design Thinking framework. Upon reflection, there are some key takeaways I‚Äôd highlight

Existing products, and their users, are a valuable resource
In developing a new feature, identifying the current product landscape is essential. There‚Äôs a lot to be learned by those who have designed similar product features. Further, the users of a similar product can be a goldmine in which to empathize with users delights and frictions.

The users experience is not confined to the app
In confronting users confusion about how this feature was distinct from other seemingly similar features, designing informational material outside the app proved beneficial. Using the medium of a website, rather than a small phone screen, gave the space to clearly display the feature use case, problems it solves, as well as how to use it. This allowed users a proper introduction to a novel feature.
